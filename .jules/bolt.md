## 2025-05-14 - Batching and Heuristic Filtering in ChromaDB Indexer
**Learning:** Performing single-item insertions into ChromaDB in a loop causes significant overhead due to repeated IPC/network calls. Additionally, CPU-intensive string operations like `.split()` can be avoided by using character-length heuristics for pre-filtering. Failing to reset batch state in a `finally` block leads to "poisoned batches" where subsequent valid items are appended to a failing batch, causing cascading failures.
**Action:** Always implement batch processing (e.g., BATCH_SIZE=50) for vector database indexing. Use `len()` as a pre-filter before expensive string manipulations. Ensure robust error handling with `try...except...finally` blocks to reset batch state and report source file ranges for easier debugging.
