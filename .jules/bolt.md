## 2025-04-18 - Batch indexing and heuristic filters for ChromaDB
**Learning:** In vector databases like ChromaDB, individual insertions incur significant network/IPC overhead. Batching reduces this overhead and allows for vectorized embedding operations. However, implementing batching requires careful nested error handling: if one file fails to read, it shouldn't discard the whole batch. Conversely, if a batch injection fails, the batch state must be cleared ("flushed") to prevent a "poisoned batch" from causing subsequent failures. Additionally, expensive string operations like `.split()` should be preceded by character-length heuristics to avoid unnecessary CPU cycles on small documents.
**Action:** Always use batch processing for database indexing. Implement nested try-except blocks: one for item preparation (continue on error) and one for batch execution (reset state in 'finally'). Use 'len()' heuristics as a high-frequency filter before expensive text segmentation.
